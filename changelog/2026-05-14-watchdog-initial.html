Initial watchdog with cooldown, backoff, and Tasmota hardening

feat 2026-05-14 · commit fe3b33a

Why

The previous /root/ont-watchdog-simple.sh on the OpenWRT router fired every two minutes whenever the default route was missing, with no cooldown. Logs showed it issuing back-to-back power cycles at 00:14 and 00:16 — before the ONT could possibly have re-synced. Separately, the Tasmota Sonoff SV was rebooting under rapid relay activity and losing its WiFi credentials, creating a deadlock where the relay would be left in the OFF state with nothing able to flip it back.

What changed

Clean-slate initial commit of the repo. Nine files:

PathPurpose
config.envSingle source of truth for hosts and timings.
router/ont-watchdog.shNew watchdog: N-of-N consecutive-fail gating, escalating cooldown, hourly cap, Tasmota liveness check.
tasmota/apply-config.shIdempotent Tasmota hardening (SetOption65, PowerOnState 1, Sleep 0, WifiConfig 4).
deploy.shOne-shot entry point: applies Tasmota, pushes to router via scp -O, rewrites crontab, runs verify.sh.
verify.shEnd-to-end health check.
README.md / AGENTS.md / CLAUDE.mdDocs + deploy-then-verify contract for any agent touching the repo.
.gitignoreStandard.

On the router, the deploy:

  • Removed the old ont-watchdog-simple.sh and ont-autoreboot.sh scripts plus their /tmp state files.
  • Installed the new watchdog and rendered /etc/ont-watchdog.conf.
  • Set cron to * * * * * /root/ont-watchdog.sh (every minute).

On the Tasmota (verified persisted via SaveCount increment):

  • SetOption65 1 — disable fast-power-cycle factory reset (protects WiFi creds).
  • PowerOnState 1 — relay defaults ON when Tasmota boots (no deadlock).
  • Sleep 0, WifiConfig 4 — keep WiFi awake, smart reconnect (weak signal at ~-80 dBm).

Watchdog decision tree

cron (every 1 min) → ping {1.1.1.1, 8.8.8.8, 9.9.9.9}
   any reply       → fail_count = 0; exit
   all fail        → fail_count += 1
                     if fail_count < 3            : exit (need ~3 min sustained failure)
                     if now - last_action < cooldown : exit (5/10/20/30 min escalating)
                     if attempts_this_hour ≥ 4    : exit (likely ISP outage)
                     if Tasmota not reachable     : exit (don't toggle a dead Tasmota)
                     → Power Off, sleep 15, Power On; update state

Verification

Ran ./deploy.sh end-to-end (idempotent); ./verify.sh exits clean. Then exercised every decision branch on the router with blackholed targets:

=== run 1/3 (expect fail=1/3) ===
=== run 2/3 (expect fail=2/3) ===
=== run 3/3 (expect Tasmota-unreachable branch) ===

state file: fail_count=3 last_action=0 attempt_count=0
log tail:
  ont-watchdog: WAN DOWN (fail=1/3)
  ont-watchdog: WAN DOWN (fail=2/3)
  ont-watchdog: WAN DOWN (fail=3/3)
  ont-watchdog: Tasmota at 192.0.2.42 unreachable; skipping toggle

The relay-toggle branch itself was deliberately not exercised live to avoid disrupting the user's internet; the Tasmota-unreachable branch immediately precedes it in the code path and proves the guard works.
