OpenWRT WiFi: switched from path-based to MAC-based radio binding (survives PCI re-enumeration)

fix 2026-05-24 · router-side change (no repo commit)

Symptom

User reported "the wireless signal is not there" again. Same surface symptom as the 2026-05-14 fix: Minion / Minion_5G not broadcasting, netifd logging Could not find PHY for device 'radio0'.

Root cause (different from last time)

The PCI bus number assigned to the mt7915e card is non-deterministic on mt7621-pci across reboots:

BootCard located atOur radio0.pathResult
2026-05-14 evening0000:01:00.00000:01:00.0✅ worked
2026-05-24 morning0000:02:00.00000:01:00.0❌ "Could not find PHY"

The previous fix's path-based binding latched onto wherever the card happened to be on that boot. It was correct in the moment but inherently fragile — any reboot could flip pcie0 / pcie1 / pcie2 enumeration ordering and break it again. OpenWRT also helpfully auto-regenerated radio2 / radio3 sections on this reboot because the original UCI sections no longer matched any present device, adding clutter.

Side effect: with both radios in retry_setup_failed, the country code (IN) couldn't be applied either, and iw reg get showed country 00 (world) — explaining why everything we tried looked back at square one.

Fix

Bind by the PHY's permanent MAC address instead of by udev path. MACs are baked into the card's EEPROM and don't move across boots:

# Per-phy permanent MACs (from /sys/class/ieee80211/phyN/macaddress)
#   phy0 (2.4 GHz only) → 3c:52:a1:24:94:af
#   phy1 (5 GHz only)   → 3c:52:a1:24:94:ae

cp /etc/config/wireless /etc/config/wireless.bak-$(date +%Y%m%d-%H%M%S)

uci delete wireless.radio0.path
uci delete wireless.radio1.path
uci set wireless.radio0.macaddr='3c:52:a1:24:94:af'
uci set wireless.radio1.macaddr='3c:52:a1:24:94:ae'

# Remove the auto-regenerated phantom radios
uci delete wireless.default_radio2; uci delete wireless.radio2
uci delete wireless.default_radio3; uci delete wireless.radio3

uci commit wireless
wifi reload

Why this won't recur

OpenWRT's wifi-device matching checks macaddr against the PHY's permanent address at startup. That value comes from the card's EEPROM (/sys/class/ieee80211/phyN/macaddress), not from PCI enumeration order, so the binding is stable regardless of which bus number the card lands on.

A related secondary benefit: with macaddr binding in place, OpenWRT's wifi-detect will not auto-regenerate phantom radio2/radio3 sections on future boots because the existing radio0/radio1 will always match.

Verification

Verified the fix on this boot. A full PCI-re-enumeration test (reboot the router and confirm WiFi returns without intervention) is left as an explicit follow-up; the user opted not to do that during the active session to avoid an unplanned outage. If it ever recurs, the entire diagnosis is reproducible via the three commands at the top of this entry.

Rollback
