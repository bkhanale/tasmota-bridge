OpenWRT WiFi: re-paired SSIDs to the real radio, set country=IN, pinned 5GHz non-DFS

ops 2026-05-14 · router-side change (no repo commit)

Superseded by 2026-05-24 — path → MAC binding. The path-based fix below worked at the time but was fragile to PCI re-enumeration on reboot.

Symptom

User reported that the OpenWRT router (10.20.30.1) was not broadcasting any WiFi. Neither Minion (2.4 GHz) nor Minion_5G (5 GHz) was visible.

Root cause

The wireless config had its SSIDs attached to phantom radios while the real card sat idle under a disabled section:

UCI sectionPath in configReality
radio0 / radio1 (enabled, SSID Minion / Minion_5G)0000:02:00.0Empty slot. Kernel logs: pcie0 no card, disable it.
radio2 / radio3 (disabled, SSID OpenWrt, no encryption)0000:01:00.0The actual mt7915e card. Confirmed via readlink /sys/class/ieee80211/phy0/device.

Netifd's response: Could not find PHY for device 'radio0' · Could not find PHY for device 'radio1' followed by Wireless device 'radio0' setup failed. The card was healthy, the kernel had detected it, the config just pointed elsewhere.

Additionally, regulatory was unset (country 00, world), which capped TX power and explained the txpower 3.00 dBm visible on the stranded managed-mode interfaces.

Fix

All changes via uci on the router, committed atomically; /etc/config/wireless backed up first to /etc/config/wireless.bak-20260514-160840.

# Backup
cp /etc/config/wireless /etc/config/wireless.bak-$(date +%Y%m%d-%H%M%S)

# Re-path the enabled radios to the real card
uci set wireless.radio0.path='1e140000.pcie/pci0000:00/0000:00:01.0/0000:01:00.0'
uci set wireless.radio1.path='1e140000.pcie/pci0000:00/0000:00:01.0/0000:01:00.0+1'

# Set regulatory domain
uci set wireless.radio0.country='IN'
uci set wireless.radio1.country='IN'

# Remove the now-redundant disabled radio2 / radio3 sections
uci delete wireless.default_radio2
uci delete wireless.radio2
uci delete wireless.default_radio3
uci delete wireless.radio3

uci commit wireless
wifi reload

After reload, 2.4 GHz came up cleanly on channel 1. The 5 GHz radio ran ACS, selected DFS channel 56, then the mt7915e driver's start_dfs_cac() failed and hostapd disabled the AP. Fixed by pinning to a non-DFS channel in UNII-1:

uci set wireless.radio1.channel='36'
uci commit wireless
wifi reload

Final state

SSIDBandChannelWidthTX powerState
Minion2.4 GHz120 MHz26 dBmAP-ENABLED
Minion_5G5 GHz36VHT8026 dBmAP-ENABLED

A station was already associated on 5 GHz shortly after reload. The Tasmota at 10.20.30.15 reconnected (incidentally to a different AP also broadcasting Minion — the home network is mesh/multi-AP, which is a useful resilience property to know about for the watchdog story).

Lessons for the watchdog

  • "Tasmota offline" can mean the AP it associates to went down, not Tasmota itself. The watchdog's Tasmota-reachability guard correctly short-circuits in this case.
  • Tasmota's mesh-driven re-association means we don't strictly need Tasmota on the OpenWRT's own SSID for the watchdog to keep working — useful when relocating the AP.

Rollback

ssh root@10.20.30.1 'cp /etc/config/wireless.bak-20260514-160840 /etc/config/wireless && wifi reload'
